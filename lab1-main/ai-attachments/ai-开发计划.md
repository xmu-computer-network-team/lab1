# 开发计划 — 可见光通信编解码器

## 总体节奏

采用迭代式开发，每个 Sprint 产出可运行的交付物，逐步叠加复杂度。

```
Sprint 0 → Sprint 1 → Sprint 2 → Sprint 3 → Sprint 4
 设计对齐    图片闭环    视频闭环    物理闭环    调优交付
  ½天         1~2天       1天        1~2天       1天
```

核心原则：**每一步都先跑通再往下走，出了问题能立刻定位是哪一层的锅。**

---

## Sprint 0 — 设计对齐（½天）

### 目标
全组理解编码方案，明确分工。

### 任务
- [ ] 所有人阅读 `ai-设计方案.md`
- [ ] 确认开发语言（Python + OpenCV）
- [ ] 确认分工（见下方分工表）
- [ ] 搭建项目骨架：目录结构、`config.py`、git 仓库
- [ ] 每人本地跑通 `pip install opencv-python numpy`

### 交付物
- 项目仓库，含目录结构和 `config.py`
- 所有人环境就绪

## Sprint 1 — 图片级闭环（1~2天）

### 目标
**编码器生成图片 → 解码器直接读图片 → 还原文件，bit 级一致。**

不涉及视频、不涉及摄像头，纯数字验证编码方案的正确性。

### 任务

| 任务 | 负责 | 说明 |
|------|------|------|
| `data_formatter.py` | A | 文件→分段+CRC-32，段→bit 列表 |
| `frame_builder.py` | A | bit 列表→帧图片（含定位图案、帧头、数据区） |
| `config.py` + `pattern.py` + `crc.py` | A | 公共模块 |
| `grid_sampler.py` | B | 直接读编码器生成的图片，按网格采样提取 bit |
| `data_recovery.py` | C | bit 矩阵→逐段 CRC 校验→还原文件 |
| 集成测试脚本 | D | 编码 1MB 文件→生成图片序列→直接解码→比对 MD5 |

### 验收标准
```bash
python encoder/encoder.py test.bin frames/
python decoder/decoder.py frames/ output.bin
md5sum test.bin output.bin   # 必须完全一致
```

### 附加：块大小标定实验（Sprint 1 期间抽 10 分钟完成）

```python
# 生成三张棋盘格测试图，分别用 6px、7px、8px 块
# 全屏显示在电脑屏幕上
# 用手机在 30~50cm 距离拍一张照片
# 人眼判断哪个尺寸还能分辨黑白格
# 结论写入 config.py 的 BLOCK_SIZE
```

**这 10 分钟决定后续所有参数，不要跳过。**

## Sprint 2 — 视频级闭环（1天）

### 目标
**编码器生成视频文件 → 解码器读视频逐帧解码 → 还原文件。**

验证视频编码（MJPG）不会破坏数据。仍然不涉及摄像头。

### 任务

| 任务 | 负责 | 说明 |
|------|------|------|
| `video_writer.py` | A/D | 帧图片序列 → MP4 视频（控制帧率和时长） |
| `frame_reader.py` | B/D | 视频 → 逐帧提取图片 |
| 命令行接口 | A | `encode in.bin out.mp4 1000` 三参数 |
| 集成测试 | 全组 | 编码→写视频→读视频→解码→比对 |

### 验收标准
```bash
python encoder/encoder.py test.bin out.mp4 5000
python decoder/decoder.py out.mp4 output.bin
md5sum test.bin output.bin   # 必须完全一致
```

### 注意
- 视频编码用 MJPG，不要用 H.264（压缩会模糊块边界）
- 如果 MJPG 出现问题，退而用 PNG 帧序列 + ffmpeg 封装

## Sprint 3 — 物理信道闭环（1~2天）

### 目标
**屏幕播放视频 → 手机拍摄 → 解码还原文件。** 这是第一次接触真实噪声。

### 任务

| 任务 | 负责 | 说明 |
|------|------|------|
| `locator.py` | B | Finder Pattern 检测 + 透视矫正（核心难点） |
| 自适应二值化 | B | 替代固定阈值，适应光照变化 |
| 置信度机制 | C | 低置信度 bit 标记为丢失 |
| 帧去重/排序 | C | 用帧序号处理重复帧和丢帧 |
| 实拍测试 | D | 不同距离、角度、光照条件测试 |

### 这个 Sprint 的典型问题和解法

| 问题 | 解法 |
|------|------|
| 找不到 Finder Pattern | 检查二值化阈值；尝试 OTSU 和 adaptiveThreshold 两种 |
| 透视变换后图像歪 | 检查四个锚点的排列顺序是否正确 |
| 大量 bit 判错 | 调整 BLACK_THRESHOLD / WHITE_THRESHOLD；改用 3×3 均值采样 |
| 部分帧完全解码失败 | 正常，拍摄时有运动模糊；统计成功率，>70% 即可 |

### 验收标准
- 手机拍摄 5 秒视频，能还原出部分正确数据
- CRC 校验通过的段，数据确实正确（零误码）
- 有效传输率 > 0（能跑通就是胜利）

---

## Sprint 4 — 调优与交付（1天）

### 目标
在 Sprint 3 跑通的基础上，优化参数，最大化有效传输率。

### 调优方向

| 参数 | 调优方法 |
|------|---------|
| BLOCK_SIZE | 如果标定实验显示 6px 可行，改 config.py 一个数字即切换 |
| FPS | 从 24 逐步试到 30，观察帧解码成功率是否下降 |
| CRC 分段大小 | 段越大效率越高但一出错丢的越多，找平衡点 |
| 二值化参数 | blockSize、C 值微调 |
| 置信度阈值 | 太高→丢失率高；太低→漏检风险。用实拍数据标定 |

### 最终交付物
- `encode` 和 `decode` 命令行工具
- 性能测试报告（有效传输率、误码率、丢失率）
- 项目文档 / 答辩 PPT

## 分工建议（3~4 人团队）

| 角色 | 主要负责 | Sprint 1 | Sprint 2 | Sprint 3 | Sprint 4 |
|------|---------|----------|----------|----------|----------|
| **A** | 编码器 + 架构 | data_formatter, frame_builder, config | video_writer, CLI | 支援调试 | 调参 |
| **B** | 解码器-定位 | grid_sampler（简单版） | frame_reader | locator（核心） | 调参 |
| **C** | 解码器-数据 | data_recovery | 集成测试 | 置信度, 帧排序 | 调参 |
| **D** | 测试 + 工具 | 集成测试, 标定实验 | 视频封装 | 实拍测试 | 报告 |

如果只有 2~3 人，A 兼 D 的工作，B 兼 C 的工作。

## 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 6px 块手机拍不清 | 中 | 传输率降低 | Sprint 1 标定实验提前暴露，回退到 8px |
| Finder Pattern 检测不稳定 | 高 | 帧解码失败率高 | 可直接移植参考代码的检测逻辑，已验证可用 |
| MJPG 压缩损失大 | 低 | bit 错误增多 | 改用无损编码或 PNG 序列 |
| 手机帧率和屏幕帧率不同步 | 中 | 出现撕裂帧 | 帧序号去重 + 失败帧直接丢弃 |
| 某人卡住进度 | 中 | 阻塞后续 Sprint | 每个 Sprint 开始时站会同步，及时调配 |
