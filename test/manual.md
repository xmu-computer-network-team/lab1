# FrameLocator 定位点识别排查与修复手册

## 1. 问题描述

`locator.py` 的 `locate_and_rectify()` 对一张手机拍屏幕的测试图片（`distorted_frame.jpg`）矫正后，输出图像相比正常方向顺时针旋转了约 30°。怀疑是定位点（Finder Pattern）找错了。

测试图片的额外困难：画面中有两块大面积镜面反光。

---

## 2. 原始架构回顾

原始流程：

```
原图 → 灰度 + 高斯模糊 → 自适应二值化
     → 1:1:3:1:1 行扫描找 Finder Pattern 候选点
     → _select_best_finders() 从候选点中选最佳 3 个
     → _order_finders() 排列为 TL / TR / BL
     → 估算 BR → 四点透视变换 → 输出矫正图
```

Finder Pattern 是类似 QR 码的同心方块定位图案，横截面的黑白比例为 1:1:3:1:1（黑-白-黑-白-黑）。扫描算法沿每一行像素滑动，维护一个 5 段状态机，当检测到符合该比例的段落时，再做垂直和水平交叉验证来确认候选点。

---

## 3. 第一轮排查：候选点数量爆炸

### 现象

运行调试脚本后发现：正常二值图上找到了 **136 个候选点**，远超预期的 3 个。

### 原因分析

`_check_finder_ratio()` 的容差 `max_dev = module * 0.8` 过于宽松。在 BLOCK_SIZE=8 的配置下，帧内部的数据区域（黑白交替的像素块）很容易碰巧满足 1:1:3:1:1 的宽松比例检查。反光区域的边缘在二值化后也会产生类似的条纹模式。

### CV 知识点：自适应二值化与噪声

`cv2.adaptiveThreshold` 使用局部窗口计算阈值，对光照不均有一定鲁棒性，但反光区域的亮度梯度会在边缘产生大量伪黑白交替，这些交替恰好能通过宽松的 1:1:3:1:1 检查。

### 修复

1. 收紧比例容差：`max_dev = module * 0.5`（从 0.8 降到 0.5）
2. 增加对角线交叉验证 `_cross_check_diagonal()`：真正的 Finder Pattern 是正方形，在对角线方向也应该满足 1:1:3:1:1 比例

### 效果

候选点从 136 个降到 **17 个**。方向正确，但仍然太多。

---

## 4. 第二轮排查：_select_best_finders 选错组合

### 现象

17 个候选点中包含了正确的 3 个点，但 `_select_best_finders()` 选出了错误的组合。

### 原因分析

原始评分公式：`score = area * rightness * ratio_score`

其中 `area`（三角形面积）权重过大，导致算法倾向于选择分布最广的 3 个点（面积最大的三角形），而不是真正的 Finder Pattern。

更致命的是：帧的宽高比为 240:135 ≈ 1.82，但透视变形后实际拍摄到的三点比例只有约 1.28。这意味着真正的 Finder 三角形的 `ratio_score` 反而很低（0.65），而一些假阳性点碰巧能组成 ratio 接近 1.82 的"完美"三角形。

### CV 知识点：透视变形对几何特征的影响

透视变换（Perspective Transform）不保持距离比和角度。一个原本宽高比 1.82 的矩形，在斜角拍摄后可能变成 1.28 甚至更极端的值。因此，**不能依赖透视变形后的几何比例来筛选定位点**。这是纯几何方法在透视场景下的根本局限。

### 尝试的修复方向

| 尝试 | 方法 | 结果 | 失败原因 |
|------|------|------|----------|
| A | 命中次数排序 + top_n 截断 | 失败 | 行扫描步长=4，每个点只被扫到1次，无区分度 |
| B | 去掉面积权重，纯 rightness × ratio_score | 失败 | 正确组合排第98/368，假阳性的 ratio 更接近期望值 |
| C | log(area) 替代 area | 失败 | 面积仍然让错误的大三角形胜出 |
| D | _finder_quality 局部质量评分 | 失败 | 帧内部数据区域也有丰富的黑白交替，质量分数无区分度 |
| E | 帧边界约束 + 最近点匹配 | 失败 | Finder 在帧内部，离边界角点的距离不可预测 |

### 关键教训

当假阳性和真阳性在特征空间中高度重叠时，单纯调参无法解决问题。需要引入新的信息源来打破僵局。

---

## 5. 第三轮排查：预处理抑制反光

### 尝试

1. **CLAHE**（自适应直方图均衡化）：`cv2.createCLAHE(clipLimit=2.0, tileGridSize=(8,8))`
2. **形态学背景归一化**：用大核闭运算估算背景光照，然后除以它
3. **Otsu 全局阈值**

### CV 知识点：CLAHE 与光照归一化

- CLAHE 在每个 tile 内做直方图均衡，能增强局部对比度，但对大面积反光效果有限（反光区域内部对比度本身就很低）
- 形态学闭运算 `MORPH_CLOSE` 用大椭圆核可以估算出平滑的背景光照分布，除以它后能有效消除渐变光照差异。这把候选点从 17 降到了 6 个

### 效果

候选点减少了，但 `_select_best_finders` 的几何评分仍然选不对——根本问题（透视变形破坏比例）没有解决。

---

## 6. 第四轮：发现帧边界检测

### 转折点

尝试用 Canny 边缘检测找帧的外边界：

```python
edges = cv2.Canny(gray_blur, 30, 100)
kernel = cv2.getStructuringElement(cv2.MORPH_RECT, (5, 5))
dilated = cv2.dilate(edges, kernel, iterations=2)
contours = cv2.findContours(dilated, cv2.RETR_EXTERNAL, cv2.CHAIN_APPROX_SIMPLE)
```

成功找到了帧的四边形边界，4 个角点坐标准确。

### CV 知识点：Canny + 轮廓近似

- Canny 边缘检测对梯度敏感，帧边界（黑色帧内容 vs 浅色背景）有强梯度，即使有反光也能检测到
- `cv2.dilate` 膨胀边缘使断裂的边缘连通
- `cv2.approxPolyDP` 用 Douglas-Peucker 算法将轮廓近似为多边形，当顶点数=4 时就是四边形
- 角点排序：用坐标和 `sum(x+y)` 和 `diff(x-y)` 可以可靠地区分四个角

### 关键发现

用这 4 个角点直接做透视变换，矫正效果已经很好——帧是正的，Finder Pattern 在正确位置。

---

## 7. 最终方案：两阶段矫正

### 思路

既然帧边界能可靠检测，而 Finder Pattern 在矫正后的图上位于已知的固定位置，那就：

1. **粗矫正**：用 Canny 检测帧边界四边形 → 透视变换到标准尺寸
2. **精矫正**：在粗矫正图上，在 Finder 的已知位置附近小范围搜索 → 精确定位中心 → 再做一次透视微调

```
原图 → Canny 边缘检测 → 找最大四边形轮廓 → 排序角点
     → 粗透视变换 → 标准尺寸图像
     → 在已知位置附近搜索 Finder 中心（1:1:3:1:1 + 交叉验证）
     → 精透视变换 → 最终输出
```

### 为什么这个方案有效

| 问题 | 原方案 | 新方案 |
|------|--------|--------|
| 假阳性过多 | 全图扫描，136→17个候选 | 只在已知位置附近小区域搜索，几乎无假阳性 |
| 选不对3个点 | 几何评分被透视变形破坏 | 不需要选择，位置已知 |
| 反光干扰 | 反光区域产生大量伪模式 | Canny 对帧边界的检测不受反光影响 |
| 透视变形 | 破坏了几何比例特征 | 粗矫正后消除了透视，精搜索在正视图上进行 |

### CV 知识点：分阶段矫正的优势

这是计算机视觉中常见的 coarse-to-fine 策略：

- 粗阶段用鲁棒但不精确的特征（帧边界轮廓）做全局定位
- 精阶段在已知范围内用精确但脆弱的特征（1:1:3:1:1 比例）做局部微调
- 两阶段互补：粗阶段消除了大部分变形，让精阶段的搜索空间大幅缩小

这个思路在很多场景中都适用：人脸检测（先找脸框再找关键点）、OCR（先找文本行再识别字符）、SLAM（先粗匹配再精对齐）。

## 8. 各方法的 CV 原理总结

### 8.1 Finder Pattern 的 1:1:3:1:1 扫描

源自 QR 码标准。Finder Pattern 的横截面在任意角度穿过中心时，黑白段的比例都是 1:1:3:1:1。扫描算法用 5 段状态机沿行滑动检测这个模式，然后用垂直、水平、对角线三个方向的交叉验证来确认。

交叉验证的作用：
- 水平扫描只能确认"这一行有 1:1:3:1:1 模式"
- 垂直验证确认"这一列也有"→ 排除水平条纹
- 对角线验证确认"对角方向也有"→ 排除非正方形的误检

### 8.2 自适应二值化

```python
cv2.adaptiveThreshold(gray, 255, cv2.ADAPTIVE_THRESH_GAUSSIAN_C,
                      cv2.THRESH_BINARY, blockSize=51, C=10)
```

- `blockSize=51`：用 51×51 的窗口计算局部阈值
- `C=10`：阈值 = 局部均值 - 10
- 优点：对光照渐变鲁棒
- 缺点：反光区域内部对比度极低，二值化结果不可靠

### 8.3 Canny 边缘检测

```python
cv2.Canny(gray_blur, 30, 100)
```

- 双阈值：强边缘 > 100，弱边缘 > 30（与强边缘相连才保留）
- 对帧边界检测效果好：帧内容与背景之间有明确的亮度跳变
- 反光不影响：反光在帧内部，帧边界的梯度仍然存在

### 8.4 透视变换

```python
M = cv2.getPerspectiveTransform(src_4pts, dst_4pts)
result = cv2.warpPerspective(image, M, (w, h))
```

- 需要 4 对对应点
- 8 自由度，能矫正任意四边形到矩形
- 精度取决于 4 个点的定位精度——这就是为什么精确的 Finder 中心比粗略的边界角点更好

### 8.5 形态学操作

```python
kernel = cv2.getStructuringElement(cv2.MORPH_ELLIPSE, (51, 51))
background = cv2.morphologyEx(gray, cv2.MORPH_CLOSE, kernel)
normalized = gray / background * 128
```

- 闭运算 = 先膨胀后腐蚀，填充暗区域中的亮细节
- 大核闭运算的结果近似于"背景光照分布"
- 除以它可以消除光照不均，但对镜面反光（饱和高光）效果有限

## 9. 失败尝试的教训

1. **调参不能解决架构问题**：当假阳性和真阳性在特征空间重叠时，无论怎么调阈值都无法分离。需要引入新的信息维度。

2. **透视场景下不要依赖欧氏几何特征**：距离比、角度、面积比在透视变换下都不保持。如果算法依赖这些特征，在正视图下工作正常，但在斜角拍摄时会失败。

3. **Coarse-to-fine 是处理复杂场景的通用策略**：先用鲁棒特征做粗定位，再用精确特征做局部微调。这比试图一步到位找到精确位置要可靠得多。

4. **问题分解比问题硬解更有效**：原始方案试图在一张透视变形+反光的图上直接找到 3 个精确的 Finder 中心——这是一个很难的问题。拆成"找边界"+"粗矫正"+"局部搜索"三个简单问题后，每一步都很可靠。
